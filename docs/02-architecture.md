# 02. 아키텍처 (Vercel + Firebase)

## 2.1 책임 경계

**원칙: 사용자 대면·렌더링·엣지 = Vercel / 상태·인증·데이터·백그라운드 잡 = Firebase.**

```
                ┌──────────────────────────── Vercel ────────────────────────────┐
   Browser ───▶ │  Next.js (App Router)                                          │
   (다크/라이트) │   ├─ RSC / UI 렌더링 (디자인 시스템)                            │
                │   ├─ Route Handlers (BFF: 세션검증, 프록시, 가벼운 집계)         │
                │   ├─ Edge Middleware (i18n 로케일 라우팅, auth 가드)             │
                │   └─ Vercel Cron (스캐너 트리거, 일배치 백테스트)                │
                └────────────────┬───────────────────────────────┬───────────────┘
                                 │ Firebase SDK / Admin SDK        │
        ┌────────────────────────▼──────────┐        ┌────────────▼───────────────┐
        │ Firebase Auth                      │        │ Firestore                   │
        │  (이메일/OAuth, custom claims=role)│        │  users/strategies/versions… │
        └────────────────────────────────────┘        └────────────┬───────────────┘
                                                                    │ 트리거
                                              ┌─────────────────────▼────────────────┐
                                              │ Cloud Functions                        │
                                              │  ├─ 전략 규칙 컴파일                    │
                                              │  ├─ 백테스트 잡 (큐/워커)               │
                                              │  └─ 스캐너 실행 (시세 데이터 소비)      │
                                              └─────────────────────┬──────────────────┘
                                                                    │
                                              ┌─────────────────────▼──────────────────┐
                                              │ 시세/차트 데이터 소스 (벤더·증권사 API) │
                                              │  ※ 라이선스·원가 미확정 (확인 필요)      │
                                              └──────────────────────────────────────────┘
```

## 2.2 왜 이 분리인가

- **Vercel**: Next.js 최적 배포·Preview·Edge·Cron. UI/i18n 라우팅·BFF에 강점.
- **Firebase**: Auth + Firestore + Functions로 상태·인증·백그라운드 잡을 한 번에. 초기 팀 규모에 운영 부담이 낮음.
- 무거운 실시간 스캔/백테스트는 요청-응답 경로(Vercel)가 아니라 **Functions/워커의 비동기 잡**으로 뺀다. → 사용자 요청은 잡을 큐잉하고 결과를 Firestore 리스너로 수신.

## 2.3 핵심 데이터 흐름

### (A) 전략 생성
1. 사용자 자연어 입력 → `AI Strategy Chat` (Route Handler가 LLM 호출).
2. `strategy-architect` 규약(→ `strategy-dsl` 스킬)으로 부족 조건 질문 → 확정.
3. 확정 전략을 **Strategy DSL(JSON)**로 저장 → Firestore `strategies/{id}/versions/{v}`.
4. Function 트리거가 DSL을 실행 가능한 Rule로 컴파일.

### (B) 백테스트
1. 사용자가 버전 지정 → 백테스트 요청.
2. Function이 잡 생성 → 과거 데이터로 실행 (**out-of-sample 분리, `backtest-guard` 준수**).
3. 결과(승률/평균수익률/MDD/평균보유기간 + 검증방식 메타)를 버전 문서에 기록.
4. UI는 Firestore 리스너로 실시간 반영.

### (C) 실시간 스캐너
1. Vercel Cron(또는 스케줄러)이 주기적으로 스캐너 Function 트리거.
2. 활성 전략들의 컴파일된 Rule을 시세 데이터에 적용.
3. **기준 캔들은 상태(state)로 유지** — 종목별로 과거 이벤트를 기억하고 현재 위치 비교 (`04-strategy-engine.md`).
4. 매칭 결과(BUY/WATCH 등)를 Firestore에 기록 → 대시보드 Live Scanner + Alerts.

### (D) 집행 (후순위)
- 검증 끝난 전략만 증권사 API 연동. **사용자 정의 조건 충족 시 주문 집행**(P1). 자체 재량 없음.

## 2.4 원가·확장성 주의

- 전 종목 실시간 스캔 + 종목별 상태 유지 = 연산·데이터 비용이 사용자 수에 비례해 증가. **유저당 원가 모델을 초기에 산정**한다.
- 완화책: (a) 전략 조건을 사전 인덱싱해 후보 종목 축소, (b) 스캔 주기 티어링(무료=지연/유료=실시간), (c) 공통 지표는 종목당 1회 계산 후 전략 간 공유.
- 시계열 대용량이 Firestore 비용을 압박하면 캔들 원본은 별도 스토어(예: 시계열 DB/오브젝트 스토리지)로 분리 검토 → `05-data-model.md`.
